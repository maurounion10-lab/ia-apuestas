El sitio se sirve desde Cloudflare Pages (no Workers puros). Los 4 Workers son servicios independientes: cron de resolución de picks, chat IA, validador de fixtures y bot de X.

Frontend (vanilla JS)

Lenguaje
JavaScript vanilla. Sin framework. Sin build step. Se sirve directo.
HTML principal (app)
index.html — ~4.5k líneas, shell de la app. (Ya no es client-index.html.)
⭐ Estilos — css/main.css
Un solo archivo (~251 KB) con 333 variables CSS. Es el corazón del sistema visual: color, tipografía, spacing y componentes salen de tokens en :root. El rediseño se hace acá.
JS externalizado (js/)
app-core.js (~732 KB, monolito — deuda) + bundles: inline, ai-chat, my-picks, footer-panel.
Theming por tokens
Existe body.wc2026-theme que se auto-activa 2-jun → 19-jul 2026 y repaletea todo (oro/rojo/verde MX) cambiando variables. Rediseñar = reescribir los tokens de :root.
📱 App Android nativa
Proyecto Android completo en android/ (Gradle) + workflow build-android.yml. Estilo Capacitor.

Backend Supabase

Proyecto: ixfrtjvhnpapyuphqfxp

URL: https://ixfrtjvhnpapyuphqfxp.supabase.co

Auth:

  • Google OAuth (con detectSessionInUrl: true).
  • Magic links con flowType: 'implicit'.

DB: Postgres. Schema parcial — hay 2 migraciones en supabase-migrations/ (se corren a mano en el Dashboard), falta el dump completo.

Tablas conocidas: acoin_users (con historial_full), shared_cache (global_historial_v1), forum_threads, forum_posts, forum_reactions.

Sync de historial en 3 capas: localStorage → acoin_users.historial_full → shared_cache (lo que ven los anónimos vía /api/sb). Un trigger Postgres (migración 2026-05-14) replica capa 2 → 3 server-side. Antes dependía del browser del admin y se quedaba "stuck".
Ojo con el flow de magic links: el default de Supabase v2 (PKCE) guarda un code_verifier en localStorage del tab que inició el flujo. Si el link se abre en otro navegador o dispositivo, el verifier no existe y la auth falla silenciosamente. Por eso se eligió implicit — pone el token en el hash.

Workers de Cloudflare (4, servicios aparte del sitio)

worker/ — cron resolver
El más importante. Cron horario que resuelve picks pendientes server-side (ESPN → TheSportsDB → manual), sin depender del browser del admin. Escribe a Supabase con service_role. Endpoint: /cron-resolve.
worker-chat/
Backend del chat IA (widget flotante).
worker-fixtures-validator/
Valida fixtures / partidos.
worker-x-bot/
Bot de publicación automática en X (@Gambeta_ai).

Deploy (Cloudflare Pages + GitHub Actions)

Hosting: Cloudflare Pages (proyecto ia-apuestas). Config en wrangler.jsonc.

Cómo deployar el sitio: push a main → deploy-pages.yml despliega solo. Nada de wrangler manual.

5 workflows en .github/workflows/:

deploy-pages
deploy-worker
deploy-worker-chat
deploy-x-bot
build-android
Post-deploy obligatorio: correr scripts/verify-deploy.sh antes de declarar "listo" (regla del CLAUDE.md del repo). Secrets CLOUDFLARE_API_TOKEN + ACCOUNT_ID viven en GitHub Actions, no en .env.
